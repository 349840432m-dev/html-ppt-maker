# 版式设计指南

逐版式的设计规范。整体视觉基因见 `08-整体视觉与细节.md`。每个版式按"结构 → 设计要点 → 动效 → 常见错误"组织；标注 【纸感】/【信息图】 表示所属预设，未标注则两者通用。

## 语义到版式的映射

选版式不是凭感觉，先判断这一页内容的**逻辑关系**，再查表：

| 内容的逻辑关系 | 版式 |
|---|---|
| 并列（N 个特点/支柱/举措） | data 卡、blueprint 卡、content 编号卡 |
| 对比（误区 vs 正解、before/after） | contrast 双栏 |
| 递进/时序（步骤、阶段、排期） | process 流程、gantt 甘特 |
| 因果/结构（要素如何支撑结论） | framework 框架、stack 堆叠 |
| 量级比较（谁大谁小） | bars 条形图 |
| 累积构成（增量如何搭成总量） | waterfall 瀑布图 |
| 条件判断（满足什么条件才做） | decision-tree 决策树 |
| 转化流失（从进入到完成损失在哪里） | funnel 漏斗诊断 |
| 机制解释（动作如何影响结果） | causal-chain 因果链 |
| 优先级判断（高低象限、分层治理） | matrix 矩阵判断 |
| 用户流程（前台动作 + 后台机制） | journey 用户路径蓝图 |
| 一个强断言 | quote 原则页 |
| 行动项/检查项 | checklist 清单 |
| 实例拆解 | case 案例页 |

**数据具象化触发规则**：内容里出现具体数值时优先转成图形而不是留在文字里——

- 1-3 个关键数字（百分比、倍数、金额）→ data 大数字卡，数字放大到正文的 4-5 倍。
- 一组可比较的数值 → bars 条形图，用长短代替数字换算。
- 有增量构成关系的数值 → waterfall。
- 带时间节点 → gantt 或 process。
- 数字没有来源 → 不触发，改用定性版式。

高级组件的完整规范见 `11-高级图表页组件.md`。公开课和培训课里，如果连续 3 页都是普通卡片/列表，必须优先改成 decision-tree、funnel、causal-chain、matrix、journey 或 before-after。

## 封面（hero）

- **结构**：kicker/chip 小标签 → 主标题（一句有立场的判断，不是名词短语）→ 副标题 → 装饰元素 → 信息脚。
- **设计要点**：
  - 标题是全 deck 最大字号（72-76px），占据视觉重心；封面禁止出现要点卡片。
  - 【信息图】签名构图：藏蓝平线 + 红点转折 + 红色上扬曲线（隐喻"从现状到突破"），底部黑色细线分隔三栏信息脚（核心主张/项目详情/发布信息）。
  - 【纸感】强排版构图：kicker 字距拉开（letter-spacing .35em）、标题下一条 72px 短横线、大量留白。
- **动效**：标题分层入场（kicker → 标题 → 副标题 → 装饰，`data-reveal` 错峰）；曲线用 `.draw` 路径绘制。
- **常见错误**：封面塞满内容；标题写成"XX 项目汇报"这种无立场的标签。

## 章节页（section）

- **结构**：巨型章节编号（背景层）+ kicker + 章节标题 + 一句 lede。
- **设计要点**：
  - 章节页的任务是制造节奏和段落感，不承载密集信息，一章一张。
  - 【纸感】深色底 + 描边巨号（340px，`-webkit-text-stroke`）+ 颗粒纹理 + 双色光晕；标题贴底部。
  - 【信息图】保持白底或极浅容器，巨号用超粗字重 + 低透明度色。
- **动效**：kicker → 标题 → lede 三层错峰。
- **常见错误**：章节页和内容页视觉强度没有差异，翻页时感受不到"进入新章"。

## 图表页总则【信息图】

适用于 bars / waterfall / gantt。三条铁律：

1. 图表占画布 ≥60%，标题贴图表。
2. 每页一层标注：箭头注释、图例或来源行。
3. 红色只标一个信息（突破口/目标/决策点），其余用藏蓝和灰分主次。

### 条形图页（bars）

- **结构**：标题 → 条形组（标签 150px 右对齐 + 轨道 + 数值）→ 红色注释行 → 黑色状态横幅。
- **设计要点**：条高 46px、间距 24px；主要项藏蓝、次要项灰、突破口红；数值加粗放条外右侧；底部黑色横幅写"当前现状"一句话结论。
- **动效**：条形按 `data-step` 分组生长（width 过渡 800ms），注释和横幅最后出现。
- **常见错误**：所有条一个颜色（丢掉主次）；数值印在条内（挤压可读性）。

### 瀑布图页（waterfall）

- **结构**：标题 → 瀑布柱（基线灰 → 增量藏蓝 → 核心增量红 → 目标柱红）→ 底部标签行 → 黑色箭头注释。
- **设计要点**：每根柱的 `--b`（起点）等于前面所有增量之和，视觉上"逐级搭上去"；总目标柱通底；数值悬在柱顶上方；左上空白区放手写式注释 + 弧线箭头指向红柱。
- **动效**：柱子按顺序 `data-step` 堆叠（height 过渡），数值随柱浮现，注释箭头最后 `.draw` 绘制。
- **常见错误**：增量柱不衔接（起点没对齐前柱顶）；忘记放注释导致"默认图表感"。

### 甘特页（gantt）

- **结构**：黑框表格（170px 行标签列 + 6 时间列）→ 藏蓝任务条 → 红菱形决策点 → 红色连接线 → 红箭头持续条。
- **设计要点**：
  - 行高 ≥104px，任务条文字两行居中；
  - 菱形 = 时间点（决策/里程碑），必须独立悬置、不贴任务条末端，加白圈描边；
  - 箭头 = 任务延伸向未来，用 `::after` 三角贴在条右端；
  - 决策点与下游任务用红色竖线连接，把因果读出来；
  - 页脚图例："◆ 关键决策点（时间点）· ▶ 持续推进（延伸至未来）"。
- **动效**：任务条按时间顺序逐条出现，菱形和连接线同步，箭头条最后。
- **常见错误**：菱形贴在条尾看起来像歪箭头；没有图例观众猜含义；条与条之间没有叙事关系。

## 框架页（framework）【纸感】/ 堆叠页（stack）【信息图】

- **framework 结构**：HTML 节点框（200×86px 白卡）+ SVG 连线层 + 中心结论圆。文字一律 HTML，SVG 只画线。
- **stack 结构**：上下两层色块（下层藏蓝、上层白）+ 横切红色虚线 + 红底红线标签 + 下方三格推进卡（黑框、▶ 分隔）。
- **设计要点**：节点先讲清楚各是什么，关系线其次，结论最后；红线标签写"硬性红线/前提条件"类的强约束。
- **动效**：节点（step 0）→ 连线绘制（step 1）→ 中心结论（step 2）。
- **常见错误**：把框架图画成装饰（节点里没有真实内容）；连线无方向感。

## 对照页（contrast）

- **结构**：标题 → 左右双栏（误区 vs 更好的做法）。
- **设计要点**：【纸感】暖橙浅底 vs 绿浅底；【信息图】红浅底 vs 绿浅底，标题带 ✕/✓ 圆形记号。两栏条目必须一一对应（坏做法一 ↔ 改造一）。
- **动效**：先揭示误区（step 0），再揭示改造（step 1），制造认知反差。
- **常见错误**：两栏内容不对仗；两栏同时出现丢掉"先错后对"的节奏。

## 数据卡页（data）

- **结构**：标题 → 三张大数字卡（数值 + 单位 + 一句解释 + 来源行）。
- **设计要点**：先写清本页唯一结论，再决定放 1、2 还是 3 个数字，不为凑三卡而添加弱数据。数值 62-64px 加粗 `tabular-nums`；数字和说明最多两级主字号，单位与数值作为一个视觉单元；**每卡必须有来源行**，没有来源就不用这个版式。默认使用中性色层级 + 一个强调色，三卡不能为了区分而各用一种鲜艳色。
- **动效**：卡片错峰入场，数值 `data-count-to` 滚动到位。
- **常见错误**：编造数字；一页同时讲多个互不相关的指标；数值和解释字号差距不够；为了“丰富”使用三种高饱和色。

## 图文并置页（image-text）

- **结构**：一侧叙事图片/真实截图 + 另一侧一句判断、简短解释和必要标注；图片与文字共享同一内容网格。
- **设计要点**：图片负责场景、证据或结构，文字负责提炼，不重复描述图片。优先使用顶边、中心线或底边中的一种明确对齐；检查型图片遵守 `data-image-role="inspect"` 和裁切契约。留白集中在标题区、图文 gutter 或页尾，不把图片缩成角落附件。
- **动效**：先出现图片建立语境，再出现判断与标注；需要讲解截图时，标注与对应区域同步揭示。
- **常见错误**：图文各自漂浮；图片比例与容器不匹配；文字比图片更抢；图片只是装饰、没有新增信息。

## 清单页（checklist）【信息图】/ 列表页（content/list）

- **checklist 结构**：居中标题 → 黑框行（92px 勾选块列 + 文字）。勾选块红蓝交替圆角方块，白色 ✓。文字里最关键的短语用红色加粗。
- **content/list 结构**：标题 + takeaway → 编号卡片/行。连续使用不得超过 3 页。
- **动效**：逐行揭示。
- **常见错误**：清单超过 4 行（该拆页）；每行长度悬殊。

## 原则页（quote）

- **结构**：巨型引号 + 一个强句子（关键词强调色）+ 出处行；【信息图】句子装进黑框圆角盒。
- **设计要点**：全页只有一句话，46-52px；没有出处就不署名，不编造。
- **动效**：引号 → 句子 → 出处三层。
- **常见错误**：句子太长（超过两行就不是"原则"）；一套 deck 超过两页 quote。

## 案例页（case-walkthrough）【纸感】

- **结构**：左侧全景画布（网格底 + 编号圆点标记）+ 右侧看点笔记卡（左红边线）。
- **设计要点**：全景用真实素材或结构示意，**禁止伪造截图**；标记和笔记编号一一对应、同步揭示。
- **动效**：标记 n 和笔记 n 同一 `data-step`。

## 流程页（process-map）【纸感】

- **结构**：编号圆点节点（描边圆）+ 标题 + 一句说明，节点间 `.draw` 连线；回流线绕行不穿文字。
- **动效**：节点推进（step n）与到达该节点的连线（step n）同步。
- **常见错误**：连线穿过文字；回流关系画成直线丢掉"闭环"感。连接线必须加 `data-connector`，并置于独立线路层；文字、节点和线路不得共用同一条绝对定位基线。

## 关系图通用线路纪律

- 先用 Grid/Flex 排好文字与节点，再建立线路层；不要先画线再把文字压在线上。
- 线路只允许经过明确留出的 gap、gutter 或轨道区，终点停在节点边界或标签前，不得从字形下方“擦过”。
- 道路图采用“文字节点区 → 短垂线 → 道路轨道 → 次级节点区”的四层结构；迁移图采用“源节点列 → 连接轨道 → 目标列表”的三列结构。
- 显式线条、道路、关系节点、连接器和箭头主干必须使用真实 DOM 元素并统一加 `data-connector`。禁止用 `::before` / `::after` 绘制承载语义的线、点或箭头；伪元素只允许承担不影响理解的纯装饰。运行 `audit_style_discipline.py` 后 `pseudo-connector` 必须为零，运行 `layout_probe.js` 后 `line-text-overlap` 必须为零。

## 蓝图卡页（blueprint）【信息图】

- **结构**：三张 2px 黑框卡：蓝图网格图示区（170px，可放线稿图标或生成图）→ 居中粗标题 → 一句定位 → 色条（红 = 最高优先级，藏蓝 = 常规）→ 行动描述。
- **设计要点**：优先级卡的色条和描述文字都用红，其余藏蓝；图示区没有素材时保留蓝图网格占位，不留纯白。
- **动效**：按优先级顺序逐卡揭示。

## 结尾页（end）

- **结构**：一句收束语（全 deck 的记忆点）+ 行动号召。
- **设计要点**：【纸感】深色底与章节页呼应；【信息图】白底巨字 + 红色粗底线（或黑框）。收束语必须是可复述的一句话，不是"谢谢观看"。
- **动效**：引导句 → 巨字 → 底线/行动项。
- **常见错误**：结尾页只有"Thanks"，浪费最后一次让观众记住结论的机会。
